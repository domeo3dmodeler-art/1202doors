# Заказ пропал / не найден — что проверить

## Почему в БД нет «истории удалений»

При удалении заказа приложение:

1. Удаляет комментарии и историю документа (`document_comments`, `document_history`) по `document_id` = id заказа.
2. Отвязывает связанный счёт (Invoice): у счёта обнуляется `order_id`.
3. Удаляет запись заказа из таблицы `orders`.

Поэтому **в БД после удаления не остаётся ни заказа, ни записей истории по этому заказу**. Узнать, кто и когда удалил, можно только по **логам приложения**.

## Как проверить на ВМ

### 1. Автоматическая проверка (с вашего ПК)

Убедитесь, что заданы переменные (или используются значения по умолчанию):

- `1002DOORS_SSH_KEY` — путь к SSH-ключу.
- `1002DOORS_STAGING_HOST` — например `ubuntu@89.169.181.191`.

Запустите:

```powershell
.\scripts\run-order-history-check-on-vm.ps1
```

Скрипт выполнит на ВМ:

- подсчёт заказов и список последних заказов;
- проверку связей заказ ↔ счёт (в т.ч. «сироты»);
- последние записи `document_history`;
- **поиск в логах** по фразам «удален пользователем», «documents/[id]/DELETE», «Заказ успешно удален».

По выводу можно понять текущее состояние заказов и, при наличии записей в логах, — кто и когда удалил документ.

### 2. Вручную на ВМ

Подключитесь по SSH и при необходимости установите клиент PostgreSQL:

```bash
ssh ubuntu@<IP_ВМ>
cd ~/domeo-app
# при необходимости: sudo apt-get install -y postgresql-client
```

Запустите проверку (скрипт из репозитория):

```bash
bash -s < /path/to/vm-order-history-check.sh
```

Или выполните SQL и поиск в логах вручную (см. блоки 1–8 в `scripts/vm-order-history-check.sh`).

### 3. Поиск по логам: кто удалил заказ

Если известен **id заказа** (например, из старой ссылки или письма):

```bash
# Логи приложения (если пишут в файл)
grep "Документ <ID_ЗАКАЗА> удален пользователем" ~/domeo-app/logs/server.log

# Или журнал systemd
journalctl -u domeo-standalone --no-pager | grep "удален пользователем"
```

В сообщении будет указан `userId` пользователя, выполнившего удаление.

Удаление заказа возможно двумя путями:

- **DELETE /api/orders/:id** — только заказ (см. `app/api/orders/[id]/route.ts`).
- **DELETE /api/documents/:id** — при выборе типа документа «order» (см. `app/api/documents/[id]/route.ts`).

В обоих случаях перед удалением вызывается `deleteDocumentCommentsAndHistory(id)`, поэтому в `document_history` записей по этому заказу не остаётся.

## Рекомендация на будущее

Чтобы сохранять факт удаления заказа (кто, когда, какой id):

- Добавить отдельную **audit-таблицу** (например, `document_deletions`: `document_type`, `document_id`, `user_id`, `deleted_at`).
- Перед вызовом `prisma.order.delete(...)` (и аналогично для других типов документов) записывать туда строку.
- Либо писать в общий лог с уровнем `info` структурированное сообщение (document_type, document_id, user_id, timestamp) и надёжно хранить эти логи (например, в отдельном файле или системе логирования).

После этого при следующем «пропаже» заказа причину можно будет восстановить по audit-таблице или по логам.
