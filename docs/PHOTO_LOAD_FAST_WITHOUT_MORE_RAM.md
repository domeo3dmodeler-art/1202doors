# Быстрая загрузка фото без увеличения RAM

См. также **docs/AUDIT_UPLOADS_AND_PERFORMANCE.md** — аудит и упрощение цепочки раздачи (без тяжёлых fallback'ов в рантайме).

## В чём проблема

502 и «подвисание» фото возникают не из‑за того, что «мало RAM», а из‑за того, что **запросы к картинкам попадают в Node**, вместо того чтобы отдаваться **Nginx с диска**.

- **Nginx** (`try_files $uri`): ищет файл по пути из URL. Если нашёл — отдаёт с диска, **Node не вызывается**, память не тратится.
- Если файл **не найден** (другой путь или имя) — запрос уходит в **Node** (`/api/uploads/...`). Node делает fallback (поиск по каталогу, readdir и т.д.). При многих таких запросах растёт нагрузка и риск OOM → процесс убит → 502.

То есть проблема не «мало оперативки», а **код и данные**: запросы не должны уходить в Node для тех файлов, которые уже лежат на диске.

## Почему Nginx не находит файл

Путь в URL приходит в нормализации **Unicode NFC** (как возвращает API). На диске же имена файлов могут быть в **NFD** (типично для Linux/macOS). Одна и та же буква «е» в NFC и NFD — разные байты, поэтому `try_files $uri` не совпадает с именем файла → Nginx идёт в Node.

## Что сделано в коде (без костылей)

1. **API отдаёт пути в NFC**  
   В `app/api/catalog/doors/complete-data/route.ts` все пути к фото из `resolveDoorPhotoToExistingFile` возвращаются в виде `.normalize('NFC')`. Браузер запрашивает по этому же пути.

2. **Однократная нормализация имён файлов**  
   Чтобы Nginx находил файлы по этому пути, имена на диске тоже должны быть в NFC. Для этого один раз нужно переименовать файлы в `public/uploads` в NFC.

## Что сделать один раз

**Локально** (в репозитории, где есть `public/uploads`):

```bash
npx tsx scripts/normalize-uploads-nfc.ts
```

Скрипт переименует все файлы в `public/uploads` в NFC (где имя отличалось от NFC). Затем синхронизируйте uploads на ВМ:

```powershell
.\scripts\sync-uploads-to-vm.ps1
# или
.\scripts\sync-final-filled-to-vm.ps1
```

После этого на ВМ в `public/uploads` будут те же имена в NFC. Nginx по `try_files $uri` будет находить файлы и отдавать их с диска, **Node для этих запросов не вызывается** — нагрузка и риск OOM падают, загрузка фото ускоряется.

**Если полная синхронизация обрывается (scp «Broken pipe»):**
- **По частям (chunked):** `.\scripts\sync-uploads-to-vm.ps1 -Subfolder "final-filled/doors" -ChunkFiles 50` — загружает по 50 файлов в одном архиве (~7–15 MB), повтор при обрыве только для текущего чанка.
- **Через rsync** (если установлен, например Git for Windows): `.\scripts\sync-uploads-to-vm.ps1 -Rsync` — инкрементальная выгрузка без одного большого архива.
- Или повторить `.\scripts\sync-uploads-to-vm.ps1 -Subfolder "final-filled/doors"` позже при стабильной сети.

## Итог

| Вопрос | Ответ |
|--------|--------|
| Нужно ли поднимать RAM, чтобы не было OOM? | Не обязательно. Достаточно, чтобы большинство запросов к фото обрабатывал Nginx с диска, а не Node. |
| Влияние кода | API отдаёт пути в NFC; в коде уже есть NFD-fallback в Node и кеш списка файлов. Оставшаяся мера — имена файлов на диске в NFC. |
| Что делать | Один раз запустить `normalize-uploads-nfc.ts` (локально), затем синхронизировать uploads на ВМ. Дальше Nginx будет находить файлы по путям из API. |

Увеличение RAM до 16 ГБ по-прежнему снижает риск OOM при пиках (PDF, много пользователей), но для **именно быстрой и стабильной загрузки фото** решающее — чтобы файлы отдавались с диска через Nginx, а не через Node.
