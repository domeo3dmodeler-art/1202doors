# Аудит: раздача фото и производительность

## Принцип

**Правильные данные один раз — простая раздача всегда.**

- Пути к фото в API должны быть **точными** и вести к файлам, которые **реально лежат** в `public/uploads/` в нормализации **NFC**. Тогда Nginx по `try_files $uri` отдаёт файлы с диска, Node не вызывается, 502 и подвисания исчезают.
- Вся «умная» подстановка (по коду ручки, по префиксу двери, по имени наличника) — это **подготовка данных** (скрипты, импорт), а не логика в каждом запросе.

## Что было не так (костыли)

| Компонент | Проблема | Решение |
|-----------|----------|---------|
| **api/uploads/route.ts** | 6 fallback’ов с readdir/findDoorPhotoFile при каждом запросе, когда файл не найден по точному пути. Нагрузка на Node, риск 502. | Раздача только по точному пути + одна попытка NFD. Остальное — 404. Подгонку путей делать при импорте/нормализации. |
| **complete-data** | Вызов findDoorPhotoFile при отсутствии файла по пути из БД — тяжёлый readdir и сопоставление на каждый запрос. | Возвращать путь из данных, нормализованный в NFC. Не вызывать findDoorPhotoFile в рантайме. Неверные пути = 404 до исправления данных. |
| **door-photo-fallback** | Используется в горячем пути (complete-data и api/uploads). Сложная логика и readdir. | Оставить только для **скриптов** (нормализация, исправление путей в БД). Убрать из complete-data и из api/uploads. |
| **ThrottledImage + image-load-queue** | Ограничение 48 одновременных загрузок — обход 503/502 при массовых запросах к Node. | При раздаче с диска через Nginx лимит не нужен в таком виде. Увеличить лимит или опираться на `loading="lazy"`. |
| **Nginx** | Неиспользуемая зона strict_limit; дублирование proxy_* в нескольких location. | Удалить мёртвую зону, упростить конфиг. |
| **Данные** | В БД/PropertyPhoto могут быть короткие имена или пути, не совпадающие с именами файлов на диске. | Однократная нормализация имён (NFC), синхронизация uploads на ВМ, при необходимости — скрипт обновления путей в БД через findDoorPhotoFile. |

## Целевая схема

1. **Фронт:** `<img src={getImageSrc(path)} />` — путь из API как есть (`/uploads/...`). Без подмены на `/api/uploads/` в браузере.
2. **Nginx:** `location /uploads/` с `root` на `public`, `try_files $uri @backend_uploads`. Файл есть — отдача с диска. Нет — проксирование в Node.
3. **Node /api/uploads/:** декодирование URL, проверка пути (только `public/uploads`), один вариант по NFD при отсутствии файла, отдача файла или 404. Без readdir по ручкам/дверям/наличникам.
4. **API (complete-data, hardware):** отдают пути в NFC к реальным файлам. Резолв «короткое имя → файл» делается при импорте/скриптах, не в запросе.
5. **Скрипты:** normalize-uploads-nfc.ts + sync-uploads-to-vm.ps1 — чтобы на ВМ имена файлов и пути из API совпадали.

## Чеклист после изменений

**Один запуск (когда есть SSH до ВМ):**

```powershell
.\scripts\run-photo-setup-full.ps1
```

Скрипт по шагам: нормализация NFC → применение Nginx на ВМ → синхронизация uploads на ВМ. Если SSH не работает (Permission denied), добавьте ключ на ВМ (Serial Console или `fix-vm-authorized-keys.ps1` с рабочим ключом), задайте `1002DOORS_SSH_KEY` при необходимости и запустите снова.

**Вручную:**

- [ ] Локально: `npx tsx scripts/normalize-uploads-nfc.ts`
- [ ] Синхронизация на ВМ: `.\scripts\sync-uploads-to-vm.ps1`
- [ ] На ВМ: путь в Nginx `root` совпадает с путём приложения (например `/home/ubuntu/domeo-app/public`)
- [ ] Применить Nginx: `.\scripts\apply-nginx-to-vm.ps1`
- [ ] При 404 по фото: путь из API должен совпадать с именем файла на диске (NFC). Если в БД/PropertyPhoto короткие или старые имена — обновить пути скриптами `scripts/match-propertyvalue-to-door-photo.ts`, `scripts/rematch-property-photos-from-doors-folder.ts` (они используют findDoorPhotoFile).
