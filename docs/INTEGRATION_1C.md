# Интеграция проекта с 1С: счёт на оплату и QR

**Только теория и исследования.** Конкретные изменения в коде не вносятся до согласования архитектуры.

Документ описывает, как организовать интеграцию с 1С для:
1. **Передачи данных счёта** из веб-приложения в 1С
2. **Создания документа «Счёт на оплату» в 1С**
3. **Получения обратно** оформленного счёта с QR-кодом для оплаты (СБП/банк)

---

## Ограничение: 1С только внутри сети компании

По требованиям безопасности **1С не может быть доступна извне** (из интернета). Сервер 1С не «смотрит наружу» и не публикуется в интернет — доступ к 1С возможен **только из внутренней сети компании**.

Следствия:
- Наше веб-приложение (если оно работает в облаке или на сервере с доступом из интернета) **напрямую вызвать 1С не может** — между ними нет сетевого пути.
- Интеграция возможна только если **звено, которое общается с 1С, работает внутри той же сети**, где стоит 1С.

Ниже варианты архитектуры с учётом этого ограничения.

---

## Варианты размещения и кто вызывает 1С

| Вариант | Где крутится наше приложение | Кто дергает 1С | Плюсы / минусы |
|--------|------------------------------|----------------|------------------|
| **1. Всё внутри сети** | Сервер приложения в офисе/ЦОД компании (в той же сети, что и 1С) | Наш бэкенд по HTTP вызывает 1С напрямую | Простая схема, один запрос «создать счёт» → ответ с PDF. Минус: пользователи и приложение должны быть доступны из этой же сети или через VPN. |
| **2. Очередь/воркер внутри сети** | Фронт и API — в облаке; отдельный **воркер или сервис** — внутри сети компании | Воркер из очереди забирает задачу «отправить счёт в 1С», сам ходит в 1С по HTTP, результат кладёт в хранилище или в БД | Приложение снаружи, 1С не светится в интернет. Задержка (асинхронность): пользователь может получать PDF по ссылке или после обновления страницы. |
| **3. Файловый обмен** | Любое | Наше приложение (или воркер внутри сети) пишет файл (JSON/XML) в **общую папку**, доступную и нашему коду, и 1С. 1С по расписанию обрабатывает файлы и кладёт ответ (номер счёта, PDF) в другую папку или в общую БД | Не нужен HTTP-сервис в 1С. Минусы: не «мгновенно», нужны права на общие каталоги, сложнее отладка и идемпотентность. |
| **4. Шлюз/прокси внутри сети** | В облаке | Внутри сети стоит **небольшой сервис-шлюз**: принимает запросы от нашего облачного бэкенда (по защищённому каналу/VPN), внутри сети ходит в 1С по HTTP и возвращает ответ | 1С по-прежнему не в интернете. Нужно поднять и поддерживать шлюз, обеспечить безопасный доступ (VPN, mTLS, токены). |

Для сценария «счёт в 1С + сразу получить PDF с QR» наиболее реалистичны:
- **Вариант 1**, если пользователи работают из сети компании или через VPN.
- **Вариант 2 или 4**, если приложение и пользователи снаружи, но есть возможность поставить воркер или шлюз внутри сети.

---

## Общая схема (при условии, что звено, вызывающее 1С, внутри сети)

```
[Пользователь] → [Наше приложение]  →  [Звено внутри сети]*  →  [1С:Предприятие]
                       |                        |                        |
                       |  Запрос «счёт в 1С»    |  1. POST данные счёта  |
                       |  (invoiceId)           |  (JSON)                 |
                       | --------------------->| ----------------------> |
                       |                        |  2. Создание документа  |
                       |                        |     «Счёт на оплату»     |
                       |                        |  3. Ответ: номер, PDF   |
                       |                        | <---------------------- |
                       |  PDF / ссылка на PDF   |                         |
                       | <---------------------|                         |
```

\* Звено внутри сети — это либо наш же бэкенд (если он стоит в сети компании), либо воркер/шлюз, который имеет доступ к 1С и к нашему приложению (очередь, API, хранилище).

- **Инициатор**: пользователь в нашем приложении (кнопка «Отправить в 1С» / «Счёт с QR»).
- **1С** создаёт документ в своей базе и возвращает номер документа и при необходимости готовый PDF с QR. Вызов 1С выполняет только компонент, работающий **внутри сети компании**.

## 1. Передача данных для счёта в 1С

### Что отправлять

Используются уже существующие в проекте сущности:

- **Invoice** (таблица `invoices`) + **cart_data** (JSON с позициями).
- В каждой позиции уже есть **sku_1c** (артикул 1С) — он заполняется из каталога (`Domeo Артикул 1С`, `Domeo_наименование ручки_1С` и т.д.).

Рекомендуемый минимальный payload для 1С (формат можно согласовать с разработчиком 1С):

```json
{
  "documentNumber": "INV-001",
  "documentDate": "2025-02-28",
  "client": {
    "name": "Иванов Иван Иванович",
    "inn": "",
    "address": "...",
    "phone": "+7...",
    "email": ""
  },
  "items": [
    {
      "sku": "артикул_1С_двери",
      "name": "Дверь межкомнатная ...",
      "quantity": 2,
      "unit": "шт",
      "price": 15000,
      "amount": 30000
    }
  ],
  "subtotal": 30000,
  "tax": 0,
  "total": 30000,
  "currency": "RUB",
  "notes": "",
  "externalId": "cuid_нашего_Invoice"
}
```

- **documentNumber** — наш номер счёта (например `invoice.number`).
- **externalId** — `invoice.id` для связи документа в 1С с записью в нашей БД.
- **items[].sku** — брать из `cart_data[].sku_1c` (уже есть в корзине и в экспорте).

Клиентские данные брать из связанной сущности **Client** (`invoices.client_id` → `clients`).

### Откуда брать данные для payload (при реализации)

- Из **Invoice** с `include: { client: true }` и разобранным `cart_data` (аналогично логике в `app/api/documents/[id]/export/route.ts`).
- Маппинг позиций: `cart_data` → `items` с полями `sku_1c` → `sku`, `name`/`model`/ручка/ограничитель — в `name`, `qty`/`quantity`, `unitPrice`/`price`, сумма по позиции.

## 2. Создание документа в 1С

Варианты реализации на стороне 1С:

- **REST/HTTP**: публикация REST-сервиса в 1С (типовой приём через HTTP-сервис или «1С:Шина»), наш бэкенд вызывает его по URL (например `POST /api/invoice/create`).
- **Обмен файлами**: наше приложение формирует XML/JSON файл в общую папку, 1С по расписанию обрабатывает и кладёт ответ (номер счёта, PDF) в папку или по callback URL. Менее удобно для «мгновенного» получения PDF с QR.

Рекомендация: **REST от нашего приложения к 1С** — один запрос «создать счёт» и в ответе получить номер документа в 1С и при необходимости ссылку на PDF или сам PDF (base64).

В 1С разработчик:
- Принимает JSON.
- Создаёт документ «Счёт на оплату» (или аналог в конфигурации).
- Заполняет реквизиты (контрагент из переданных данных или поиск по ИНН/имени), табличную часть по артикулам (`sku`).
- Формирует QR для оплаты (СБП/реквизиты) по правилам ЦБ/банка.
- Возвращает в ответе: `{ "success": true, "documentNumber1C": "...", "pdfBase64": "..." }` или URL для скачивания PDF.

## 3. Получение счёта с QR-кодом для оплаты

Варианты:

| Вариант | Описание |
|--------|----------|
| **A** | 1С в ответ на «создать счёт» сразу отдаёт PDF (с уже вшитым QR). Мы сохраняем файл и отдаём пользователю. |
| **B** | 1С возвращает только номер документа и реквизиты (ИНН, счёт, назначение платежа, сумма). Мы сами генерируем QR (например СБП) и подкладываем в свой шаблон счёта или в готовый PDF. |
| **C** | Отдельный запрос к 1С: «выдать PDF по номеру документа» — 1С отдаёт готовый PDF с QR. |

Для «правильно оформленного счёта» с точки зрения бухгалтерии предпочтительно **документ создаётся в 1С**, а клиенту выдаётся тот же документ в виде PDF. Поэтому оптимально: **создание документа в 1С + получение PDF от 1С** (варианты A или C). Если 1С не отдаёт PDF, можно вариант B с генерацией QR у нас (по реквизитам от 1С).

QR для оплаты в РФ обычно:
- **СБП** — строка в формате, который банки/ЦБ публикуют для QR-оплаты (формируется в 1С или у нас по реквизитам).
- **Обычный платёж** — реквизиты (назначение, счёт, ИНН, КПП, сумма) + при необходимости мы можем сгенерировать QR из этих данных.

При реализации в проекте можно выделить:
- Модуль интеграции (клиент к 1С, маппинг Invoice → payload, вызов «создать счёт» и при необходимости «получить PDF»).
- API-эндпоинт, принимающий `invoiceId`, инициирующий отправку в 1С (напрямую или через очередь), сохранение номера документа 1С в БД и возврат клиенту PDF или ссылки на скачивание.

## 4. Возможная структура при реализации

При переходе к коду можно вынести интеграцию, например, в:

- Модуль с типами payload/ответа 1С, маппингом Invoice → JSON для 1С, HTTP-клиентом к 1С (или постановкой задачи в очередь для воркера внутри сети).
- API-маршрут, который по `invoiceId` запускает этот процесс и отдаёт пользователю PDF или ссылку.

Переменные окружения (идея): URL 1С (используется только звеном внутри сети), учётные данные, флаг включения интеграции. В БД — поле для номера документа в 1С по соответствующему счёту (идемпотентность и связь документов).

---

## 5. Последовательность шагов при реализации

1. **Определить архитектуру по ограничению «1С только внутри сети»**: где будет крутиться звено, вызывающее 1С (наш сервер в офисе, воркер внутри сети, файловый обмен, шлюз), и как оно будет получать задачи от облачного приложения (очередь, VPN, общая папка).
2. **Согласовать с разработчиком 1С** формат JSON (поля, названия), способ аутентификации, точные URL или правила файлового обмена (создание счёта, получение PDF).
3. **Реализовать в 1С** приём данных, создание документа «Счёт на оплату», формирование QR и выгрузку PDF (в ответе HTTP или в файле ответа).
4. **В нашем проекте**: маппинг Invoice → payload, вызов 1С (из компонента внутри сети), сохранение номера документа 1С в БД, API и UI для инициации и получения PDF.
5. **Идемпотентность**: сохранять в БД номер документа 1С по `invoice.id` и при повторном запросе не создавать дубликат в 1С (использовать `externalId`).

После этого передача данных для счёта в 1С, создание документа в 1С и получение обратно оформленного счёта с QR будут организованы в одном контуре интеграции с учётом доступа к 1С только из внутренней сети.
